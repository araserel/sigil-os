# Glossary

Plain-English definitions for terms used throughout Prism documentation.

---

## Workflow Terms

| Term | Definition |
|------|------------|
| **Specification (Spec)** | A written description of what a feature should do — the "what" without the "how" |
| **Constitution** | Your project's rulebook defining standards all work must follow (accessibility, security, etc.) |
| **Track** | Workflow path based on complexity: Quick (small fixes), Standard (typical features), Enterprise (large initiatives) |
| **Phase** | A stage in the workflow: Assessment → Specify → Clarify → Plan → Tasks → Implement → Validate → Review |
| **Handoff** | A package of information passed between workflow stages or to engineers for review |

---

## Priority & Status Terms

| Term | Definition |
|------|------------|
| **P1 / P2 / P3** | Priority levels: P1 = must have, P2 = should have, P3 = nice to have |
| **Blocker** | Something preventing progress until resolved |
| **Human checkpoint** | A decision point that pauses the workflow until you approve |

---

## Technical Terms

| Term | Definition |
|------|------------|
| **API endpoint** | A URL where your application receives or sends data (like a mailbox address for software) |
| **Lint / Format** | Automated tools that check and fix code style — like spell-check for code |
| **QA loop** | An automated fix-and-validate cycle that runs up to 5 times before asking for help |

---

## Prism-Specific Terms

| Term | Definition |
|------|------------|
| **Agent** | An AI specialist that handles a specific type of work (e.g., Business Analyst, QA Engineer) |
| **Skill** | A reusable capability an agent can invoke (e.g., write a spec, validate code) |
| **Chain** | A sequence of skills that run together to complete a workflow |
| **Orchestrator** | The central agent that routes your requests to the right specialist |
